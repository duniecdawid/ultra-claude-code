Execution

The execution engine that runs any plan through per-task pipeline teams. This is the single source of truth for how plans are executed.

For architecture context, see Architecture. For component reference, see Components.

Overview

Plan Execution reads the entire plan directory (documentation/plans/{name}/) and runs it through per-task pipeline teams. On trigger, the Lead reads ALL files in the plan directory — README.md, any existing tasks/ files, shared/ directory (if resuming), checkpoint files — to build complete context before spawning teams.

Prerequisites:

  • CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS: "1" in the plugin’s settings.json (project-level setting applied by Claude Code when the plugin is active — not a plugin.json field)
  • tmux installed
  • Plan exists in documentation/plans/{name}/README.md

Agent teams only (per decision D1 and D13).

The Coordination Model

Three mechanisms work together:

MechanismPurpose
Per-Task Pipeline TeamsLead spawns a dedicated mini-team for each task. Each team has its own Executor, Reviewer, and Tester, plus access to a shared Tech Knowledge agent for external library docs. The team self-coordinates internally through all pipeline stages.
Shared Directory (plans/{name}/shared/)Contains lead.md with plan overview, concurrency decisions, architectural constraints, and key decisions. Written by the Lead. Survives session death.

Per-Task Artifact Files (plans/{name}/tasks/task-N/)Each task gets a subdirectory with plan.md and impl.md — persistent context written by the task’s pipeline team.

Context Bridge — How Task Teams Get Context

Task team members do NOT inherit the Lead’s conversation history. They start with a blank context window + project CLAUDE.md (auto-inherited). The Lead bridges the gap by including these paths in every team’s spawn prompt:

Team MemberReceives in Spawn Prompt
All team membersPlan README.md path, shared/lead.md path, task description from plan
Task Executor+ architecture docs path (documentation/technology/architecture/), coding standards path (documentation/technology/standards/), drives the review/test cycle via SendMessage. Can spawn Explore subagents for codebase investigation and query the shared Tech Knowledge agent for external library docs.
Code Reviewer+ architecture docs path, coding standards path, SendMessage target (Executor) for feedback
Task Tester+ success criteria from plan, .claude/system-test.md (if exists), SendMessage target (Executor) for feedback

Example spawn prompt (Task Executor):

“You are a Task Executor for Task 3: Add user authentication middleware.
Read these files for context:
- Plan: documentation/plans/user-auth/README.md
- Lead notes: documentation/plans/user-auth/shared/lead.md
- Architecture: documentation/technology/architecture/
- Standards: documentation/technology/standards/

Write your implementation plan to tasks/task-3/plan.md. Write implementation notes to tasks/task-3/impl.md. Drive the review/test cycle by sending messages to your Reviewer and Tester teammates when ready. You have 10 fix cycles before escalating to Lead.”

Per-Task Pipeline

Each task flows through pipeline stages. The Lead creates tasks with stage metadata:

Task stages: pending → planning → impl → review → test → done

How it flows:

  • Lead creates all tasks as pending
  • Lead spawns a per-task team for each task (respecting concurrency limits)
  • Within each team, the Executor drives the pipeline:
    • Executor plans and implements → writes tasks/task-N/plan.md and tasks/task-N/impl.md (spawns Explore subagents for codebase investigation, queries Tech Knowledge agent for external library docs as needed)
    • Executor sends “ready for review” to Reviewer
    • Code Reviewer evaluates → PASS or FAIL with feedback to Executor
    • Executor sends “ready for test” to Tester
    • Task Tester runs tests → PASS or FAIL with feedback to Executor
  • On FAIL: Executor fixes code, updates impl.md, re-triggers review/test (max 10 fix cycles)
  • On PASS (both review and test): team reports “task done” to Lead and exits
  • After 10 fix cycles: Executor escalates to Lead with full context

Review and test run in parallel after the Executor signals readiness. Both must PASS before the task is complete.

Task Pipeline

Every task gets the full pipeline team: Executor + Reviewer + Tester + Tech Knowledge (3 agents per task + 1 shared knowledge agent). There is no classification step — all tasks receive the same treatment.

PipelineTeam
Planning → Implementation → Review → TestExecutor + Reviewer + Tester + Tech Knowledge (shared)

Each task must deliver end-to-end testable user value — a tester verifies by simulating user behavior, not checking technical artifacts. Trivial work (config changes, renames, one-liners) is absorbed into the nearest task — never listed as a standalone task.

As each team completes its task and exits, the Lead spawns the next pending task’s team. Teams are independent — no cross-team coordination needed.

Model selection per role (these values correspond to the model: field in each agent’s .md frontmatter — see Components):

RoleModelRationale
Tech KnowledgesonnetShared agent for external library docs — lightweight, on-demand queries
Task ExecutoropusWriting production code is the highest-stakes task — quality here reduces downstream review/test failures and retry loops

Task TestersonnetPer-task test execution + final full test suite gate

Note: Agent model and tools are declared in each agent’s .md frontmatter. Spawn prompts from the Lead provide additional runtime context (plan paths, task description, shared directory) on top of the base agent definition.

Required Permissions

Teammates inherit Lead’s permission mode. For unattended execution, pre-approve in settings:

  • Bash: test runners, build commands, linters
  • Write/Edit: source files, test files, documentation
  • Read/Glob/Grep: always allowed (read-only)

Task Tester is read-only for source code — it should NOT have Write/Edit on src/. Code Reviewer is fully read-only (Write, Edit, and Bash are all disallowed). Tech Knowledge agent is read-only.

Shared Directory

plans/{name}/shared/ contains Lead-level shared context:

plans/{name}/shared/
  lead.md            # Lead writes here -- plan overview, constraints, decisions

What goes in lead.md:

  • Plan overview and concurrency decisions
  • Architectural constraints discovered during execution
  • Key decisions made by Lead (e.g., task re-ordering, amendments)
  • Cross-task dependency notes (“Task 3 created UserService — Task 5 should import from src/services/user.ts”)

Per-task context lives in task subdirectories:

plans/{name}/tasks/
  task-1/
    plan.md          # Executor writes: implementation plan
    impl.md          # Executor writes: implementation notes, integration points
  task-2/
    plan.md
    impl.md
  ...

Why this structure: Each task team is independent. Per-task files eliminate cross-team write conflicts. The Lead writes only to shared/lead.md; each task team writes only to its own tasks/task-N/ directory.

Lead Intervention Points

Lead does NOT micromanage. Intervenes only at:

  1. Team spawning — decides concurrency, writes spawn prompts, creates shared/ directory with initial lead.md, creates tasks/ subdirectories
  2. Task completion tracking — monitors which task teams have finished, spawns next pending task’s team
  3. Escalation handling — receives escalations from Executors after 10 failed fix cycles, decides whether to adjust task or escalate to user
  4. Checkpoint — periodic state save (see below)
  5. Plan-invalidating discovery — if a task team discovers something that fundamentally changes the plan (e.g., an API doesn’t exist, a dependency is incompatible), the Executor sends a message to Lead. Lead evaluates impact and either adjusts tasks or escalates to user.
  6. Completion synthesis — collects results, produces summary
  7. Team shutdown — ensures all teams have exited, cleans up

Lead behavior: Acts as a project manager — minimal intervention, data-driven decisions, escalates to user when uncertain. Spawn prompts are direct, structured, and context-rich. Does not micromanage task implementation; trusts task teams to self-coordinate internally.

Between these points, task teams self-coordinate via their internal pipeline (Executor drives review/test cycles via SendMessage).

Checkpoint Architecture

Progress is persisted in two places:

WhatWhereWhen
Task states and Lead decisionsplans/{name}/shared/lead.mdUpdated by Lead as tasks complete, decisions are made
Per-task artifactsplans/{name}/tasks/task-N/{plan,impl}.mdWritten by task teams during execution (already on disk)

Checkpoint triggers:

  • Every 3 completed tasks
  • User runs /uc:checkpoint
  • Before risky plan amendments

Checkpoint file (plans/{name}/checkpoint-{timestamp}.md) captures a snapshot:

  • All task states (pending/in-progress/done with current pipeline stage)
  • Active task teams and their current assignments
  • Decisions made during execution
  • Blockers encountered
  • Files modified so far

On resume: Lead reads plan README.md + latest checkpoint + shared/lead.md + all files in tasks/ → knows exactly what’s done vs pending → spawns new task teams for remaining work → skips completed tasks.

Error Recovery

Failure ClassDetectionRecovery
Task failure (review/test fails)Reviewer or Tester sends FAIL to Executor within the task teamExecutor fixes code, re-triggers review/test (max 10 fix cycles). After 10 failures: Executor escalates to Lead with full context. Lead decides whether to adjust task or escalate to user.
Team crash (session crash, context overflow)Lead detects stalled progress for a task teamLead spawns a replacement team for the task. Per-task artifacts (tasks/task-N/) preserve context — new team reads existing plan/impl and continues.
Session death (everything dies)User reruns /uc:plan-execution {plan-name}Lead finds latest checkpoint, shows resume prompt. New teams spawned, but task states + shared/lead.md + tasks/ artifacts preserve all progress.

Mid-Execution Plan Changes

When a discovery invalidates part of the plan:

  1. Task Executor sends urgent message to Lead with evidence of the invalidation
  2. Lead evaluates scope of impact:
    • Single task affected — Lead updates task description, restarts the task team if needed
    • Multiple tasks affected — Lead adds amendment to shared/lead.md, updates affected task descriptions, may cancel pending task teams
    • Plan fundamentally wrong — Lead escalates to user with evidence. User decides: amend plan or abort and re-plan.
  3. Lead resumes spawning after resolution

Session Resume

When /uc:plan-execution fires for a plan that has existing checkpoint files, the plan-execution skill handles resume as its first action:

  1. Reads latest checkpoint-{timestamp}.md file in the plan directory
  2. Reads shared/lead.md for Lead-level context
  3. Reads all per-task artifact files in tasks/ directory
  4. Shows user a resume prompt with progress summary
  5. Skips completed tasks and spawns new task teams for remaining work

This is skill startup behavior, not a hook.

Workflow: Step-by-Step

User: /uc:plan-execution user-auth

PHASE 1: SETUP (Lead actions)
  a. Read ENTIRE plan directory:
     - documentation/plans/user-auth/README.md (plan + tasks)
     - documentation/plans/user-auth/tasks/*/ (existing artifacts)
     - documentation/plans/user-auth/shared/lead.md (if resuming)
     - documentation/plans/user-auth/checkpoint-*.md (if any)
     Lead now has full picture of plan state.
  b. Check for checkpoint:
     - If checkpoint exists -> show resume prompt
     - If resuming -> skip completed tasks based on checkpoint
  c. Create shared/ directory with initial lead.md
  e. Create tasks/task-N/ subdirectories for each task
  f. Determine concurrency (1-3 tasks: 2, 4-8: 3, 9+: 4 max teams)

PHASE 2: PIPELINE ORCHESTRATION (per-task teams, self-coordinating)

  For each task (up to concurrency limit):
    Lead spawns a per-task agent team:

    Executor:
      read shared/lead.md + architecture docs + standards ->
      spawn Explore subagents for codebase investigation ->
      query Tech Knowledge agent for external library docs ->
      write tasks/task-N/plan.md ->
      implement code ->
      write tasks/task-N/impl.md ->
      send "ready for review" to Reviewer
      send "ready for test" to Tester

    Code Reviewer (parallel with Tester):
      check quality, patterns, architecture ->
      PASS: signal Executor | FAIL: feedback to Executor

    Task Tester (parallel with Reviewer):
      run tests, check criteria ->
      PASS: signal Executor | FAIL: feedback to Executor

    If FAIL (either):
      Executor fixes, re-sends for review/test (up to 10 cycles)
    If PASS (both):
      Team reports "task done" to Lead, team exits

  As teams complete, Lead spawns next pending task's team.

PHASE 3: CHECKPOINT (triggered by Lead)
  Triggers: every 3 completed tasks,
            user /uc:checkpoint,
            before risky plan amendments
  Saves: task states, checkpoint file, decisions
  Per-task artifacts already on disk

PHASE 4: FAILURE HANDLING (within teams + Lead escalation)
  Review/test failure: Executor fixes, re-triggers (team-internal)
  Max 10 fix cycles -> Executor escalates to Lead
  Lead decides: adjust task, reassign, or escalate to user
  Team crash: Lead spawns replacement team, per-task files preserved
  Session death: user reruns /uc:plan-execution, reads checkpoint

PHASE 5: COMPLETION (Lead actions)
  a. All task teams have reported done
  b. Lead asks a Tester to run full test suite (final gate)
     - Not per-task -- entire suite as regression check
     - If fail: Lead decides whether to re-queue or report
  c. Produce summary: tasks, files changed, decisions, tests
  d. Present summary to user
