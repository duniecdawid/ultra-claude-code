Workflows

Step-by-step flows showing how each mode operates end-to-end. For the architecture behind these flows, see Architecture.

Invocation: All skills are namespaced under the plugin name uc (from plugin.json). Full invocation names are /uc:feature-mode, /uc:debug-mode, /uc:doc-code-verification-mode, /uc:discovery-mode, /uc:plan-execution. See Components for the complete skill table.

Feature Mode

Entry condition: User has a feature idea or requirement to implement.
Exit condition: Plan approved and ready for execution, or user decides not to proceed.

User: /uc:feature-mode "Add user authentication"

1. SCOPE CHALLENGE
   → Feature Mode activates, challenges scope and pushes for clarity
   → Asks "why?", ensures requirements are specific

2. CONTEXT GATHERING
   → Code Surveyor + Doc Surveyor subagents scan codebase and documentation in parallel
   → Researcher subagent gathers deeper context (architecture, product docs, codebase)
   → Docs Manager skill loaded for documentation awareness

3. DOCUMENTATION UPDATE
   → Architecture decisions written to documentation/technology/architecture/
   → If architecture decision is ambiguous: suggests RFC mode
     → Creates documentation/technology/rfcs/{NNN}-auth-strategy.md
     → Runs AI persona review (Devil's Advocate, Pragmatist, etc.)
     → Outcome integrated into architecture doc, RFC archived

4. PLAN CREATION
   → Plan Enhancer scaffolds plan directory
   → Plan created in documentation/plans/user-auth/README.md (task list embedded)
   → Every task gets the full pipeline (Researcher + Executor + Reviewer + Tester)

5. PLAN REVIEW
   → User reviews and approves plan
   → If user rejects plan:
     → User provides feedback on what to change
     → Mode re-enters planning with feedback as additional context
     → Revised plan generated and re-presented
   → If user partially rejects:
     → User specifies in chat which tasks to remove or modify
     → Plan updated in place, task list adjusted
     → Re-presented for final approval

6. EXECUTION
   → /uc:plan-execution user-auth
   → Per-task pipeline teams run tasks from the plan (see Plan Execution below)

Edge cases:

  • RFC mode disagreement — If AI personas reach no consensus, Lead presents all perspectives and user decides.
  • Scope creep during planning — If research reveals the feature is much larger than expected, Claude flags this and suggests splitting into multiple plans.
  • Missing architecture docs — If architecture docs don’t exist yet, Feature Mode creates them as part of the Documentation Update phase.

Plan Execution

See Execution for the complete 5-phase workflow.

Summary: /uc:plan-execution {plan-name} → Lead reads plan → spawns per-task pipeline teams (Researcher, Executor, Reviewer, Tester for every task) → each team self-coordinates internally through pipeline stages → up to 2–4 teams run concurrently → checkpoints at every 3 tasks, on user request, and before risky amendments → Tester runs final gate → summary report.

Discovery Mode

Entry condition: User wants to research a topic without writing code.
Exit condition: Research findings written to documentation, available as context for future planning.

User: /uc:discovery-mode "Research how competitors handle rate limiting"

1. STRATEGIC SCOPING
   → Discovery Mode skill activates (Head of Product persona)
   → Coding is DISABLED
   → Defines: problem statement, who cares, what decisions, artifact plan

2. PARALLEL RESEARCH
   → Researcher + Market Analyzer subagents spawned in parallel via Task tool
   → Researcher: Explores internal codebase, reads docs, uses Ref.tools
   → Market Analyzer: Web searches, competitor analysis, user insights

3. SYNTHESIS
   → Lead collects findings from both subagents
   → Reconciles contradictory findings, identifies gaps
   → Distills raw research into actionable insights

4. DOCUMENTATION
   → Artifacts produced to documentation/product/ as needed:
     → research/{topic}.md — always produced (raw findings)
     → description/{topic}.md — always produced (distilled brief)
     → requirements/{topic}.md — when building something
     → personas/{persona-name}.md — when user context matters

5. SUMMARY
   → Presents key findings, recommendations, and next steps to user
   → No code changes. No plan. Pure investigation output.
   → Feeds into future planning sessions as context.

Edge cases:

  • No relevant results found — Agents report what they searched and suggest alternative angles or broader/narrower search terms.
  • Contradictory findings — Researcher and Market Analyzer may find conflicting information. Both perspectives are documented with sources, user decides which to prioritize.
  • Scope too broad — If the topic is too large for a single discovery session, Lead suggests breaking it into focused sub-topics.

Init Project

Entry condition: User wants to onboard a project to Ultra Claude.
Exit condition: Project scaffolded with documentation structure, coding standards, and system-test.md.

User: /uc:init-project

1. SAFETY CHECK
   → Warns about directory creation and documentation moves
   → Waits for user confirmation

2. EXPLORE
   → Code Surveyor + Doc Surveyor subagents scan codebase and docs in parallel
   → Code Surveyors detect standards signals (API patterns, auth, error handling, etc.)
   → Code Surveyors catalog test infrastructure (frameworks, commands, CI steps)
   → Asks user about anything ambiguous

3. PLAN
   → Groups 1-4: scaffold, configure, migrate, regenerate index
   → Group 2.5: Standards & System Test Approval
     → Presents candidate standards table with evidence strength
     → Per-topic options: Create / Skip / Create (I'll add context)
     → System-test.md outline for approval
     → User must explicitly approve before researchers spawn

4. EXECUTE (three-stage pipeline)
   → Stage A (parallel): Core scaffolding — Task Executors handle Groups 1-4
   → Stage B (parallel with A): Researchers gather evidence per approved standard + system-test
   → Stage C (after B): Task Executors write standards + system-test.md from research
     → Standards use prompt-architect methodology (persona, context, few-shot, chain-of-thought)
     → Each standard follows templates/standard.md structure
     → Research files deleted after standard is written

5. SUMMARY
   → Reports: scaffolded dirs, configured files, migrated docs
   → Lists created standards with core principles
   → Notes system-test.md security categories and tester rules
   → Flags thin-evidence standards for review

Edge cases:

  • No standards signals detected — If Code Surveyor finds no recognizable patterns, Group 2.5 is skipped and no standards are generated. System-test.md is still produced if test infrastructure is found.
  • User skips all standards — Stages B and C for standards are skipped entirely. Only system-test.md research/writing runs if approved.
  • Thin evidence — Standards with < 3 concrete rules include a NOTE section about limited evidence. User is prompted to add examples post-init.

Doc & Code Verification Mode

Entry condition: Codebase and documentation exist; user wants to check consistency.
Exit condition: Discrepancy plan approved and ready for execution, or no discrepancies found.

User: /uc:doc-code-verification-mode

1. SCOPE DETERMINATION
   → Doc & Code Verification Mode activates
   → Determines scope (full project or specific directory if argument provided)

2. STRUCTURAL SURVEY
   → Spawns Code Surveyor + Doc Surveyor subagents in parallel via Task tool
   → Code Surveyors scan codebase structure, patterns, APIs
   → Doc Surveyors scan documentation claims, specs, architecture docs
   → Builds verification matrix of doc-code pairs

3. CROSS-REFERENCE CHECK
   → Checker subagents compare doc claims vs code reality (spawned in parallel)
   → Each Checker handles a specific doc-code pair from the matrix

4. SYNTHESIS
   → Discrepancies listed with severity (Critical/Major/Minor)
   → Grouped by theme, prioritized by impact

5. FIX PLANNING
   → Plan Enhancer scaffolds plan directory
   → Plan created with fix tasks (update doc or update code)
   → Note: the verification skill handles its own scaffolding via Plan Enhancer

6. PLAN REVIEW
   → User reviews and approves plan
   → If user rejects plan:
     → User provides feedback on what to change
     → Mode re-enters planning with feedback as additional context
     → Revised plan generated and re-presented
   → If user partially rejects:
     → User specifies in chat which tasks to remove or modify
     → Plan updated in place, task list adjusted
     → Re-presented for final approval

7. EXECUTION
   → /uc:plan-execution {plan-name}
   → Per-task pipeline teams resolve discrepancies per plan

Edge cases:

  • No discrepancies found — Mode reports clean status. No plan created.
  • Ambiguous discrepancy — When it’s unclear whether code or docs are “correct,” the discrepancy is flagged for user decision rather than auto-resolved.
  • Very large codebase — Surveyors may hit context limits. Mode supports scoped verification (e.g., /uc:doc-code-verification-mode src/auth/ for a specific directory).

Debug Mode

Entry condition: User has a bug or issue to investigate.
Exit condition: Fix plan approved and ready for execution, or issue determined to be external/not reproducible.

User: /uc:debug-mode "Login fails intermittently on staging"

1. ANALYSIS
   → Debug Mode skill analyzes the issue description
   → Identifies affected components, potential areas of concern

2. STRUCTURAL SURVEY + HYPOTHESES
   → Code Surveyor + Doc Surveyor subagents scan relevant code and docs
   → Proposes hypotheses based on survey findings (e.g., race condition, session expiry, cache stale)

3. PARALLEL INVESTIGATION
   → Spawns Researcher subagents — one per hypothesis, investigating in parallel via Task tool
   → System Tester subagent attempts to reproduce the bug
   → Evidence gathered, hypotheses ranked by likelihood

4. EVIDENCE SYNTHESIS
   → Collects all investigation results
   → Ranks hypotheses by evidence strength

4.5. DOCUMENTATION UPDATE
   → Updates architecture or standards docs if investigation reveals undocumented behavior
   → Mirrors Feature Mode's documentation update phase

5. FIX PLANNING
   → Plan created focused on the fix + verification steps
   → Plan Enhancer scaffolds plan directory

6. PLAN REVIEW
   → User reviews and approves plan
   → If user rejects plan:
     → User provides feedback on what to change
     → Mode re-enters planning with feedback as additional context
     → Revised plan generated and re-presented
   → If user partially rejects:
     → User specifies in chat which tasks to remove or modify
     → Plan updated in place, task list adjusted
     → Re-presented for final approval

7. EXECUTION
   → /uc:plan-execution {plan-name}
   → Per-task pipeline teams implement the fix
   → System Tester validates the fix resolves the original issue

Edge cases:

  • Bug not reproducible — System Tester reports inability to reproduce. Lead presents findings and asks user for more context (environment details, logs, steps to reproduce).
  • Multiple root causes — Investigation reveals the symptom has multiple contributing causes. Plan includes fixes for all identified causes, ordered by impact.
  • Fix requires architectural change — If the fix would violate existing architecture, Debug Mode flags this and suggests running Feature Mode instead for a proper design review.
